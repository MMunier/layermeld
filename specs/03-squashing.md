# 03 — Squashing

Each input image carries an ordered stack of layers. Before the tool
can reason about cross-image deduplication, every input image is
collapsed to one *logical* filesystem: the state that would be visible
to a process inside a freshly started container based on that image.

Squashing is performed entirely in-memory as an index over tar entries;
no bytes are written to disk during this phase.

## 3.1 The squashed-filesystem index

For each input image the tool builds a `SquashedFs` map:

```
path -> SquashedEntry {
    image_id:   InputImageId,
    layer_idx:  usize,           // which layer the surviving entry came from
    entry_idx:  usize,           // index within that layer's tar stream
    kind:       File | Dir | Symlink | Hardlink | Char | Block | Fifo,
    mode:       u32,             // permission bits + setuid/setgid/sticky
    uid:        u64,
    gid:        u64,
    size:       u64,             // for regular files; 0 for everything else
    xattrs:     BTreeMap<String, Vec<u8>>,
    link_target:Option<PathBuf>, // for symlinks/hardlinks
    rdev:       Option<(u32,u32)>,// for char/block devices
    // mtime is NOT stored — see 06-timestamp-normalization
}
```

The `image_id`/`layer_idx`/`entry_idx` triple is enough to re-open the
exact tar entry on the second pass without unpacking anything.

## 3.2 Apply order

Layers are applied bottom-up in the order given by the image's manifest
(`manifest.layers[0]` first, `manifest.layers[n]` last). Within a
single layer, entries are applied in the order they appear in the tar
stream.

For each entry of layer `i`:

- **Whiteout** `dir/.wh.name`: remove `dir/name` from the index, and if
  `dir/name` was a directory, recursively remove everything underneath
  it. The whiteout marker itself is not stored.
- **Opaque directory** `dir/.wh..wh..opq`: remove every existing entry
  whose path is a strict descendant of `dir`. The marker itself is not
  stored. Entries for `dir` itself are preserved (the opaque marker
  hides children, not the directory).
- **Regular entry**: insert/overwrite the index entry for that path.
  An overwrite for a directory replaces the directory's metadata but
  leaves its existing children untouched, matching overlayfs semantics.

A whiteout can only target a path that existed in a *lower* layer of
the same image. A whiteout for a path that does not exist in the
current squashed view is silently dropped (it is a no-op, not an
error).

## 3.3 Hardlink resolution during squashing

Hardlinks are normalized into the index as `Hardlink { link_target }`
entries. Their identity for deduplication purposes (04-file-identity)
is determined by the file the link points at: a hardlink is never
itself a candidate for the shared layer, but its target may be.

If a hardlink's target falls outside the squashed view (because a
later layer whited the target out while keeping the link), the link is
demoted to a regular file: its body bytes are read from the original
target's tar entry and re-attached to the link's path. This is the
only case where a hardlink "becomes" a regular file.

## 3.4 What is dropped

The following are discarded by squashing and never appear in any
output layer:

- All intermediate file states from layers below the topmost surviving
  one.
- All whiteout markers and opaque-dir markers.
- The `history` array of the original image config (a fresh single-
  entry history is generated by 08-image-config-and-manifest).
- `created_by` strings, `comment` strings, and any layer-level
  `created` timestamps from the inputs.

## 3.5 What is preserved

- Every surviving file's body (referenced by `(layer_idx, entry_idx)`
  for later streaming).
- Mode bits including setuid/setgid/sticky.
- Uid and gid as raw numeric values. Username/groupname strings from
  PAX `uname`/`gname` headers are dropped — only the numeric ids are
  meaningful for runtime.
- All `user.*`, `security.*`, and `trusted.*` xattrs carried in PAX
  `SCHILY.xattr.*` headers.
- Symlink targets and device major/minor numbers verbatim.

## 3.6 Per-image config carry-over

The OCI image config (env, entrypoint, cmd, working dir, exposed
ports, labels, user, stop signal, healthcheck, etc.) of each input
image is preserved untouched on its corresponding output image. Only
the parts that describe the layer history and timestamps are
rewritten — see 08-image-config-and-manifest.
